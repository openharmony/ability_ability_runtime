# 高影响缺陷检查清单（HIGH_IMPACT_CHECKLIST）

范围：`ability_base` 仓全模块。适用于 C++、ArkTS/TS、Rust、构建脚本、资源处理、IPC/序列化、文件/路径、压缩包/二进制解析等所有会影响模块服务稳定性和安全性的代码。

目标：**避免任何会对该模块提供的服务产生影响的缺陷**，按问题最终可能导致的影响进行优先级排序，而不是仅凭代码异味判断。


## 优先级模型

对每个候选问题使用以下公式：

```text
优先级 = 影响严重程度 * 可触发性 * 影响范围
```

| 优先级 | 结果影响 | 典型原因 | 修复紧迫性 |
| --- | --- | --- | --- |
| P0 | 进程崩溃、卡死、死锁、非法内存访问、OOM、未捕获的 Native 异常 | 空指针解引用、越界访问、释放后使用、无限循环、阻塞主线程、跨越 Native/IPC 边界抛异常 | 立即 |
| P1 | 数据损坏、错误的破坏性副作用、权限绕过、持续性资源泄漏、状态污染 | 错误的路径写入/删除、句柄/线程/引用泄漏、生命周期状态损坏、不安全的全局状态 | 高 |
| P2 | 功能失败、错误结果、可恢复的边界错误、平台不兼容 | 边界情况处理不当、错误码不一致、不支持的输入 | 计划 |
| P3 | 可观测性、可维护性、低影响的体验问题 | 日志不清晰、命名 confusing、实现重复 | 暂缓 |

每个发现都必须回答：

- 这可能导致什么结果？
- 谁或什么可以触发它？
- 一旦被触发，影响范围有多大？

## 执行分层与排查路径

这份清单是全仓通用母版，不应为每个模块单独改写。实际排查时按三层使用：先跑所有模块都适用的核心问题，再按模块能力选择专项条目，最后用历史问题模式做补充对照。

### 第一层：必扫核心清单

每个模块都必须先确认以下问题，优先找 P0/P1：

- 是否存在空指针、越界访问、释放后使用、未初始化数据使用。
- 是否存在死循环、递归无上限、死锁、主线程或关键线程阻塞。
- 是否存在未捕获异常、`bad_alloc`、OOM、异常跨 Native/IPC/API 边界。
- 是否存在外部输入驱动的反序列化、字符串解析、数组/对象解析、类型转换。
- 是否存在 size、offset、index、count 的溢出、下溢、截断或负数转 unsigned。
- 是否存在资源泄漏，包括 fd、mmap、线程、socket、timer、callback、native reference。
- 是否存在状态机半成功、错误路径未回滚、close/dispose 后继续使用。
- 是否存在返回值被忽略、错误被吞掉、未知类型默认成功。
- 是否能从调用点证明问题可由 UI、API、IPC、配置、文件、网络或测试入口触发。

### 第二层：能力专项清单

按模块实际能力选择阶段 6 的专项类别，不需要每个模块逐条执行所有类别：

| 模块能力 | 优先专项 |
| --- | --- |
| 文件、路径、压缩包、外部二进制 | 类别 1、2、3、5、8、9 |
| Parcel、JSON、字符串、参数容器 | 类别 2、4、5、7、9 |
| mmap、fd、socket、native handle | 类别 2、3、6、7 |
| 全局缓存、单例、异步、并发读写 | 类别 3、4、6 |
| 权限、安全校验、代码/资源加载 | 类别 5、7、8、9 |
| 纯数据转换、工具函数 | 类别 4、5、7、9 |

### 第三层：历史问题模式库

阶段 6 中的典型反例来自已审计模块，作用是帮助快速联想同类缺陷。它不是要求每次全量打勾，而是在发现相似代码结构时反查：

- 是否有相同结果影响。
- 是否有相同触发条件。
- 是否有相同修复模式。
- 是否需要把新问题沉淀为新的通用检查项。

### 推荐排查节奏

1. **建立模块风险画像**：每个模块用一句话描述主要能力和最高风险，例如“参数容器/反序列化模块，高风险是空输入、嵌套结构、类型转换和 OOM”。
2. **选择最高风险入口**：每个模块先挑 3-5 个入口，例如公开 API、反序列化入口、文件/网络读取入口、close/dispose、全局缓存访问。
3. **按结果影响提问**：这个入口能不能导致 crash、hang、OOM、数据破坏、权限绕过、资源耗尽或状态污染。
4. **确认触发路径**：高优先级候选必须能写成“通过 API Y，给定输入 X，代码路径 Z 可导致结果 R”。
5. **按证据分级输出**：区分已确认问题、强疑似问题、可疑但待验证问题、已排除问题，避免把所有代码味道都混成高危。
6. **用测试闭环**：对 P0/P1 至少补充一种验证方式，如单测、畸形输入、fuzz、ASan/UBSan、TSan、并发压力或大 size/OOM 用例。

### 证据等级

| 等级 | 含义 | 输出要求 |
| --- | --- | --- |
| Confirmed | 有明确代码证据和可触发路径 | 进入正式发现，给出修复和测试建议 |
| Likely | 代码风险明确，但触发路径还需补充确认 | 作为高优先级候选跟进 |
| Suspicious | 只是模式可疑，结果或触发性不明确 | 放入 follow-up，不阻塞主线 |
| Excluded | 已确认被上游条件、不变量或调用约束保护 | 记录排除理由，避免重复排查 |

## 阶段 1：暴露面梳理

对每个模块记录：

| 模块 | 公共 API | C++ 入口 | 主要能力 | 是否异步 | 资源所有权 | 副作用 | 初始风险 |
| --- | --- | --- | --- | --- | --- | --- | --- |

优先关注具有以下能力的模块：

- 文件、路径或目录操作
- 进程、命令或 shell 执行
- 网络、下载或上传
- 数据库或本地存储
- 权限、令牌或凭据处理
- 系统信息或环境访问
- 线程、异步任务、回调、监听器或观察者
- Native 句柄或引用所有权
- 不可信二进制/压缩包/外部格式解析
- IPC 序列化/反序列化（Parcel、JSON、字符串）

## 阶段 2：P0 崩溃与卡死扫描

### 空指针解引用

搜索模式：

- `ptr->member`
- `*ptr`
- `ptr[index]`
- `map.find(...)->second`
- `dynamic_cast` / `Query` / `Get` 后直接使用
- 字符串/数组 `[0]`、`front()`、`back()` 在空输入上的使用

检查问题：

- 指针是否可能来自查找失败、转换失败、分配失败或无效的外部 id？
- 每次解引用前是否都检查了 `nullptr`？
- 上层传入的 `null`、缺失字段或无效 id 是否可能到达该路径？
- 空字符串、空数组是否在索引前被拦截？

### 越界访问

搜索模式：

- `vec[i]`、`arr[i]`、`str[i]`
- `data + offset`
- `memcpy`、`memmove`、`strcpy`、`strncpy`
- 在 Native 边界未捕获异常时使用 `at()`

检查问题：

- 索引是否在所有路径上都与 `size()` 做了校验？
- 负值是否可能变成巨大的 `size_t`？
- `offset + length` 是否可能溢出？
- 在使用 `[0]` 前是否处理了空字符串或空数组？
- 括号/引号匹配函数返回值是否在 substr 前校验？

### 释放后使用与生命周期

搜索模式：

- `delete`、`free`、原始指针所有权
- 异步回调捕获 `this` 或原始指针
- 监听器/观察者持有 Native 对象引用
- JS/Native 对象生命周期绑定

检查问题：

- 回调是否可能在所属对象销毁后仍执行？
- `close`、`dispose` 或 `unsubscribe` 是否能完全阻止后续回调？
- 重复释放是否会导致同一资源被 double-free？
- `Close`/`Dispose` 后是否阻止后续操作再使用已释放资源？

### 卡死、死锁、主线程阻塞

搜索模式：

- `while (...)`、`for (;;)`、解析循环
- `join`、`wait`、`sleep`
- `mutex`、`condition_variable`、锁嵌套

检查问题：

- 每个循环分支是否都会推进或退出？
- 每个等待是否都有超时或保证信号？
- 主线程上是否可能执行阻塞 IO 或长时间计算？
- 是否会在持锁时调用外部代码？
- 循环/递归解析是否限制次数或深度？

### 未捕获的 Native 异常

搜索模式：

- `throw`
- `std::stoi`、`std::stol`、`std::filesystem`、JSON 解析
- `optional.value()`、`variant get`、`vector::at`
- `make_unique`、`std::string::resize` 等大分配

检查问题：

- 异常是否会跨越 Native 或 IPC 边界？
- 转换失败是否被转换为稳定的错误返回？
- `at()` 和解析异常是否在入口点附近被捕获？
- 开启 `-fexceptions` 的模块是否对 `std::bad_alloc` 做入口级防护？

## 阶段 3：P1 副作用与资源扫描

### 破坏性副作用

检查问题：

- 用户或上层输入是否能控制目标路径、id 或资源？
- 路径遍历、符号链接跟随或平台路径差异是否可能逃逸预期范围？
- 失败时是否会留下部分写入或损坏的状态？
- 安全校验函数是否存在 shortcut/early return 绕过核心检查？

### 资源泄漏

检查项：

- 文件描述符
- `mmap` 区域
- 套接字
- 线程
- 定时器
- Native 引用
- JS 回调引用
- 异步任务句柄
- 监听器/观察者

检查问题：

- 每次成功获取后，是否在成功和错误路径上都有释放？
- 泄漏是否是每次调用都会重复发生？
- 重复调用是否会导致 OOM、句柄耗尽或线程耗尽？
- 不同模式/类型的对象生命周期规则是否一致？

### 状态污染

检查问题：

- 全局映射、缓存和 id 注册表是否会被清理？
- close/dispose 状态是否是幂等的？
- 并发读写是否受到保护？
- 错误路径是否会恢复部分已更改的状态？
- 成功/打开标志是否只在所有子步骤成功后设置？

## 阶段 4：跨边界契约扫描

仅关注可能演变为 P0/P1 的契约问题。

检查问题：

- `null`、`undefined`、缺失字段和无效枚举在 C++ 中会变成什么？
- 数值转换是否可能溢出或丢失精度？
- `size_t` 转换为较小类型前是否校验范围？
- unsigned 减法是否可能下溢？
- 乘法/加法表达式是否可能溢出？
- 字符串编码对于路径、id 和序列化数据是否安全？
- 数组和缓冲区长度是否有界？
- 转换失败是否会阻止执行？
- 缺失字段是否可能变成危险的默认值？
- 未知/未识别类型是否默认返回失败而非通过？

## 阶段 5：调用点确认

对每个 P0/P1 候选问题，确认：

- 该 API 是否可从 UI、插件、配置、IPC、测试或用户可控输入到达？
- 调用方是否在调用 Native 代码前验证输入？
- 调用是否可能重复、竞态或在 close/销毁后发生？
- 调用方是否忽略错误或使用部分结果继续执行？

目标结论应具体：

```text
通过 API Y，给定输入 X，代码路径 Z 可导致崩溃/卡死/数据丢失。
```

## 阶段 6：按能力归纳的高影响模式检查

以下 9 类模式是对阶段 2–4 的细化，适用于 extractortool、want 及同类 Native 子模块的快速扫描。

### 类别 1：不可信二进制/外部文件格式解析

适用场景：Zip/HAP/APK/ELF/任意自定义二进制、图片/音视频头、网络协议包、Parcel。

| # | 检查项 | 检查方法 / 搜索模式 | 典型反例 |
| --- | --- | --- | --- |
| 1.1 | 所有从文件读取的“大小”字段都必须有业务上限 | 搜索 `compressedSize`、`uncompressedSize`、`sizeOfCentralDir`、`nameSize`、`extraSize`、`commentSize`、`length`、`bufferSize` | 直接按 `zipEntry.uncompressedSize` 分配输出缓冲区 |
| 1.2 | 偏移 + 长度必须做溢出安全校验 | 检查 `offset + length`、`startPos + bufferSize`、`nextOffset` 等表达式 | 32 位中间结果溢出后再读取 |
| 1.3 | 所有 magic/signature 必须在读取后续字段前校验 | 搜索 `signature`、`LOCAL_HEADER_SIGNATURE`、`EOCD_SIGNATURE` | `ReadLocalHeaderName` 读完头却不校验签名 |
| 1.4 | 循环/递归解析必须限制次数或深度 | 搜索 `for (; ;)`、`while`、递归调用 | `IsEntryDataConsistent` 遍历本地头无次数上限 |
| 1.5 | 字段长度必须与实际可读字节数一致 | 检查 `remainingSize`、边界比较 | 中央目录声明的大小大于实际缓冲区 |
| 1.6 | 不要对未知/保留类型默认通过 | 搜索 `default:`、`else return true` | 未知压缩方法、未知数组类型直接返回 true |
| 1.7 | 对截断/特殊值（如 0）保持警惕 | 检查 `== 0` 的 shortcut | `localHeaderOffset == 0` 直接视为一致 |

核心口诀：**先校验元数据，再按元数据读取；元数据本身也可能在说谎。**

### 类别 2：大内存分配与异常安全

适用场景：任何基于外部输入大小分配堆内存、字符串、vector、数组的地方。

| # | 检查项 | 检查方法 / 搜索模式 | 典型反例 |
| --- | --- | --- | --- |
| 2.1 | 基于外部大小的分配优先使用 `new (std::nothrow)` 或 `std::make_unique<T[]>` + try/catch | 搜索 `new uint8_t[`、`new char[`、`resize(`、`std::make_unique<.*\[` | `std::make_unique<uint8_t[]>(zipEntry.compressedSize)` 未捕获 `bad_alloc` |
| 2.2 | 为单条目和总体设置内存上限 | 检查是否有 `maxAllowedSize`、阈值常量 | 无上限读取中央目录或条目数据 |
| 2.3 | 避免一次性分配完整输出，优先流式/分块 | 检查解压/解码是否使用固定大小缓冲区 | 一次性分配完整解压后数据 |
| 2.4 | 异常路径必须释放已分配资源 | 检查 `try/catch`、RAII、失败分支的 cleanup | `InitZStream` 失败后未释放 bufOut |
| 2.5 | 拷贝构造/拷贝赋值必须满足强异常安全 | 检查是否先 `clear()` 再分配 | `WantParams::operator=` 先清空再拷贝大 unsupported 数据 |
| 2.6 | 析构函数必须释放本类拥有的所有资源 | 检查 `~Class`、成员释放 | `SAFE_ABC` FileMapper 不 `munmap` |

核心口诀：**大分配要防 OOM，异常路径要回滚，资源归属要清晰。**

### 类别 3：资源生命周期与所有权

适用场景：mmap、文件描述符、文件指针、native 句柄、缓存中的 shared_ptr、全局单例。

| # | 检查项 | 检查方法 / 搜索模式 | 典型反例 |
| --- | --- | --- | --- |
| 3.1 | 每个 `mmap` 必须有对应 `munmap` | 搜索 `mmap`、`munmap`、`autoReleaseMem` | SAFE_ABC 映射未设置释放标志 |
| 3.2 | 每个文件描述符/指针必须在所有路径关闭 | 搜索 `fopen`、`open`、`fd_`、`fclose`、`close` | 异常分支未 fclose |
| 3.3 | 不同模式/类型的对象生命周期规则要一致 | 检查枚举类型处理分支 | SHARED_MMAP 自动释放，SAFE_ABC 不释放 |
| 3.4 | 共享缓存（cache map）必须明确对象所有权 | 检查 `shared_ptr` 缓存、引用计数 | Extractor 缓存 map 中 Extractor 生命周期依赖 shared_ptr |
| 3.5 | Singleton/全局状态访问要加锁或 atomic | 检查 `static` 成员读写 | `ExtractResourceManager` 无锁读写 shared_ptr |
| 3.6 | `Close`/`Dispose` 后必须阻止后续操作再使用已释放资源 | 检查 `isOpen_`、reset 顺序 | `ZipFile::Close` 与并发操作存在 race |

核心口诀：**谁分配谁释放，谁持有谁负责；共享资源要同步，关闭要彻底。**

### 类别 4：状态机与部分失败

适用场景：打开/关闭、初始化、拷贝赋值、解析半失败、并发状态切换。

| # | 检查项 | 检查方法 / 搜索模式 | 典型反例 |
| --- | --- | --- | --- |
| 4.1 | 成功/打开标志只能在所有子步骤成功后设置 | 检查 `isOpen_ = true`、返回 result 的顺序 | `ZipFile::Open` 在解析失败后仍置 `isOpen_ = true` |
| 4.2 | 失败时不能留下半初始化/半打开对象 | 检查失败路径是否清理部分状态 | 解析失败后的 ZipFile 仍被当作已打开 |
| 4.3 | 拷贝/赋值使用 copy-and-swap 或先完成所有分配再替换 | 搜索 `clear()` 后 `=`、`new` 在 `clear` 之后 | `WantParams::operator=` 弱异常安全 |
| 4.4 | 对未知/失败类型不能静默丢弃数据 | 搜索 `continue`、`return true` 在 else 分支 | `NewParams` 丢弃未知类型和失败数组 |
| 4.5 | 嵌套解析返回值必须检查 | 搜索 `->PublicReadFromParcel(`、`->ReadFromParcel(` 等调用 | `ReadFromParcelWantParamWrapper` 忽略返回值 |

核心口诀：**要么全成功，要么全回滚；不能要成功标志不要成功状态。**

### 类别 5：字符串与路径解析

适用场景：文件路径、URI、模块名、zip 条目名、URL、JSON key。

| # | 检查项 | 检查方法 / 搜索模式 | 典型反例 |
| --- | --- | --- | --- |
| 5.1 | 路径解析后必须校验是否逃逸白名单目录 | 检查 `realpath` 后是否有前缀校验 | `MakeNewJsModulePath`、`MakeFilePath` 无最终目录前缀校验 |
| 5.2 | 处理前导/后缀/连续分隔符、空字符串 | 测试 `"/"`、`"dir/"`、`"//"`、`""` | `GetChildNamesCache` 未去尾 `/` |
| 5.3 | 使用 `realpath` 等函数时缓冲区大小要正确 | 检查 `PATH_MAX`、`resize`、数组/vector 使用 | `ZipFileReader::init` 用 `resize(PATH_MAX - 1)` 给 realpath |
| 5.4 | 静态类型转换前要做动态检查 | 搜索 `static_cast<.*Parcel*`、`static_cast<.*MessageParcel*` | `WriteToParcelFD` 裸 `static_cast<MessageParcel*>` |
| 5.5 | 字符串/数组解析必须校验空输入和最小长度 | 搜索 `[0]`、`front()`、`back()`、`substr(` | `Array::Parse` 未检查空字符串就索引 `[0]` |
| 5.6 | 括号/引号匹配函数返回值必须校验 | 搜索 `FindMatchingBrackets`、`find` 返回值用于 substr | `ParseWantParamsWithBrackets` 未检查 `-1` |

核心口诀：**路径要圈范围，字符串要验边界，转型要查类型。**

### 类别 6：并发与同步

适用场景：全局缓存、单例、打开/关闭、读写共享状态。

| # | 检查项 | 检查方法 / 搜索模式 | 典型反例 |
| --- | --- | --- | --- |
| 6.1 | 全局 map/cache 读写必须加同一把锁 | 检查 `unordered_map`/`map` 与 `mutex` 配对 | `ExtractorUtil::GetExtractor` 首次读未加锁 |
| 6.2 | double-checked locking 必须使用 atomic | 检查 `if (cache.find(...))` + lock + 再次检查 | `ExtractorUtil::GetExtractor` 非原子 DCL |
| 6.3 | `Close`/`Dispose` 必须与并发操作互斥 | 检查 `isOpen_` 读写是否原子 | `ZipFile::Close` 与 Extract 操作可能 race |
| 6.4 | shared_ptr 全局对象读写要加锁或 atomic | 检查 `std::shared_ptr` 成员读写 | `ExtractResourceManager` 无锁 shared_ptr |

核心口诀：**共享状态要同步，关闭要上锁，DCL 要原子。**

### 类别 7：返回值与错误码

适用场景：任何可能失败的 API，尤其是 Parcel/流/文件/网络读取。

| # | 检查项 | 检查方法 / 搜索模式 | 典型反例 |
| --- | --- | --- | --- |
| 7.1 | 所有可能失败的读取 API 返回值必须检查 | 搜索 `parcel.Read`、`stream.read`、`ReadBuffer` 调用 | `ReadFromParcelString` 不检查 `ReadString` 返回值 |
| 7.2 | 嵌套解析/反序列化返回值必须检查 | 搜索 `Unmarshalling(`、`PublicReadFromParcel(` | `ReadFromParcelWantParamWrapper` 忽略返回值 |
| 7.3 | 错误路径不能返回 true 或部分成功 | 搜索 `return true` 在错误分支 | `WriteMarshalling` 对未知类型返回 true |
| 7.4 | 未知类型/默认值必须显式处理，不能 fallthrough 为成功 | 搜索 `default:`、`else` | `ReadArrayToParcel` 对未知数组类型返回 true |
| 7.5 | 流写入必须检查状态或捕获异常 | 搜索 `ostream::write`、`ofstream::write` | `Extractor::GetFileBuffer` 直接 write 不检查 |

核心口诀：**失败要报错，不能假成功；返回值要检查，不能吞异常。**

### 类别 8：安全校验深度防御

适用场景：任何带有“安全”、“校验”、“一致”字样的函数，以及从外部加载代码/配置/资源的入口。

| # | 检查项 | 检查方法 / 搜索模式 | 典型反例 |
| --- | --- | --- | --- |
| 8.1 | 安全校验函数不能有任何 shortcut/early return | 检查 `if (xxx == 0) return CONSISTENT` | `IsEntryDataConsistent` 对偏移 0 直接通过 |
| 8.2 | 对压缩/加密/特殊格式条目要明确拒绝或严格校验 | 检查 `compressionMethod`、`flags` 处理 | `GetSafeData` 对 data descriptor 仍返回映射 |
| 8.3 | 一致性校验必须覆盖所有关键字段 | 检查 crc、size、offset、name 是否都校验 | `CheckCoherencyLocalHeader` name 长度校验逻辑缺陷 |
| 8.4 | 对未知/未识别类型默认应返回失败而非通过 | 搜索 `default:` 返回 true | 未知接口类型直接 `return true` |
| 8.5 | 安全加载路径要拒绝压缩后的可执行数据 | 检查 `.abc`、`.so` 等加载入口 | SAFE_ABC 映射到压缩条目仍继续 |

核心口诀：**安全校验不能偷懒，未知即危险，默认要拒绝。**

### 类别 9：整数运算与数值边界

适用场景：任何涉及大小、偏移、索引、计数的算术运算。

| # | 检查项 | 检查方法 / 搜索模式 | 典型反例 |
| --- | --- | --- | --- |
| 9.1 | `size_t` 转换为较小类型（`int32_t`、`int`）前要校验范围 | 搜索 `static_cast<int>`、`static_cast<uint32_t>` | `DoMarshalling` 把 `size_t` 写入 `WriteInt32` |
| 9.2 | 减法表达式在 unsigned 类型上要防止下溢 | 检查 `a - b` 其中 `a`、`b` 为 unsigned | `fileLength_ - length` 潜在下溢 |
| 9.3 | 乘法/加法表达式要防止溢出 | 检查 `offset + size`、`len + adjust`、`nameSize + extraSize` | `GetLocalHeaderSize` 相加前未校验 |
| 9.4 | 返回类型要能容纳实际值 | 检查函数返回类型与成员类型是否匹配 | `FileMapper::GetOffset` 返回 `int32_t` 但 `offset_` 是 `size_t` |

核心口诀：**unsigned 减法会下溢，大转小会截断，相加相乘要溢出检查。**

## 评审输出规则

- 按优先级排序发现，而不是按文件顺序。
- 在可触发性被证伪之前，将 P0/P1 候选视为高优先级。
- 仅在结果影响或可触发性明显有限时才降级。
- 将已确认的发现与后续跟进问题分开。

## 使用建议

1. **新模块首次审计**：先走“执行分层与排查路径”，完成模块风险画像、核心必扫项和最高风险入口确认；再按模块能力选择阶段 6 的专项类别。
2. **全仓专项扫描**：按能力横向扫，而不是按目录机械扫。例如先扫所有反序列化入口，再扫所有文件/路径入口，再扫所有全局缓存/并发入口。
3. **代码评审**：只要 PR 涉及外部输入、反序列化、资源生命周期、并发状态、路径/文件、权限/安全校验，就必须引用本清单对应条目说明风险已被处理。
4. **问题输出**：正式报告优先放 P0/P1 的 Confirmed 和 Likely；Suspicious 单独作为 follow-up，避免稀释修复优先级。
5. **回归测试**：为确认问题补充能证明结果影响的用例，如大 size、空字符串、非法类型、截断元数据、嵌套递归、并发 Close、重复释放、未知类型。
