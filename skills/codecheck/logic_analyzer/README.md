# Logic Analyzer Skill - 使用指南

## 简介

Logic Analyzer 是一个专门用于**深度分析代码逻辑变更**的技能。与常规代码审查不同，本技能专注于：

- 🔍 **逻辑影响范围分析**：理解代码修改会影响哪些功能
- 🐛 **逻辑问题发现**：发现隐藏的逻辑错误和缺陷
- ✅ **状态转换验证**：确保状态机转换的正确性
- 🚨 **边界条件检查**：发现边界条件处理不当的问题
- 📊 **业务规则验证**：确保业务逻辑规则的正确性

## 核心功能

### 1. 三层分析模型

```
第一层: 变更识别 (What Changed)
  ↓
第二层: 逻辑分析 (Logic Analysis)
  ↓
第三层: 影响评估 (Impact Assessment)
```

### 2. 七大分析维度

| 维度 | 检查内容 | 发现的问题 |
|------|----------|------------|
| **控制流** | 分支、循环、条件 | 死代码、逻辑矛盾、覆盖不完整 |
| **数据流** | 变量定义、赋值、使用 | 未初始化、数据污染、类型不匹配 |
| **状态机** | 状态转换、转换条件 | 非法转换、状态不一致、死锁 |
| **边界条件** | 数组边界、空值、极值 | 越界、空指针、溢出 |
| **错误处理** | 错误码、异常、清理 | 遗漏错误路径、资源泄漏 |
| **并发控制** | 锁、原子操作、竞态 | 死锁、竞态条件、数据竞争 |
| **业务规则** | 约束、不变性、契约 | 规则违反、不变性破坏 |

## 问题类型详解

### 控制流问题

#### 🔴 死代码 (Dead Code)
永远不会执行的代码段，可能由以下原因造成：
- 永不为真的条件
- return/break后的代码
- 重复的检查

#### 🟠 逻辑矛盾 (Logic Contradiction)
条件之间存在矛盾，导致某些路径永远无法执行：
- 互斥条件同时为真
- 冗余的条件检查
- 条件覆盖矛盾

#### 🟡 条件覆盖不完整
枚举、switch-case缺少某些分支的处理

---

### 数据流问题

#### 🔴 未初始化变量
使用未初始化的变量，导致未定义行为

#### 🔴 数据污染
外部数据未经验证直接使用，可能导致：
- 缓冲区溢出
- 命令注入
- SQL注入

#### 🟠 类型不匹配
有符号/无符号比较、枚举类型混用

---

### 状态机问题

#### 🔴 非法状态转换
跳过必要的中间状态，破坏状态机完整性

#### 🟠 状态不一致
多个状态变量不同步，导致读取不一致

#### 🟠 状态机死锁
无法到达终态，造成系统挂起

---

### 边界条件问题

#### 🔴 数组越界
索引未验证、边界计算错误

#### 🔴 空指针解引用
未检查空指针就直接使用

#### 🔴 整数溢出
算术运算导致溢出，产生意外结果

---

### 错误处理问题

#### 🔴 遗漏错误路径
不检查返回值，错误被忽略

#### 🔴 资源泄漏
错误路径未释放资源，导致内存/句柄泄漏

---

### 并发控制问题

#### 🔴 死锁
锁顺序不一致、循环等待

#### 🔴 竞态条件
Check-Then-Act模式、非原子操作

#### 🔴 数据竞争
无保护的并发访问共享变量

---

### 业务规则问题

#### 🔴 不变性破坏
操作破坏类/模块的不变性

#### 🔴 契约违反
前置/后置条件未验证或保证

## 分析检查清单

### 使用方法

1. **选择检查维度**：根据代码变更类型选择相关维度
2. **执行检查**：按照检查清单逐项验证
3. **记录问题**：发现问题时记录详细信息
4. **评估影响**：分析问题的影响范围和严重程度
5. **提供建议**：给出具体的修复方案

## 严重等级说明

| 等级 | 说明 | 示例 | 是否阻塞上库 |
|------|------|------|-------------|
| 🔴 **致命** | 必须修复，否则影响系统正确性 | 非法状态转换、死锁、内存泄漏 | ✅ 是 |
| 🟠 **严重** | 强烈建议修复，可能导致问题 | 竞态条件、错误路径遗漏 | ✅ 是 |
| 🟡 **警告** | 建议修复，不影响基本功能 | 冗余代码、不完整的错误处理 | ⚠️ 视情况 |
| 🟢 **建议** | 可选优化，改进代码质量 | 代码风格、性能优化 | ❌ 否 |

## 最佳实践

### 1. 分析前准备

```bash
# 确保在正确的分支上
git checkout your-feature-branch

# 确保代码已编译通过
./build.sh

# 运行现有测试
./test.sh
```

### 2. 分析时注意

- 📌 **关注变更的上下文**：理解为什么这样修改
- 📌 **考虑所有可能路径**：不要遗漏边界条件
- 📌 **验证假设**：不要假设变量总是有效
- 📌 **检查并发场景**：多线程环境下的行为
- 📌 **验证业务规则**：确保符合业务约束

### 3. 分析后处理

- ✅ **创建修复issue**：为每个致命/严重问题创建issue
- ✅ **添加单元测试**：覆盖发现的问题场景
- ✅ **更新文档**：记录重要的逻辑变更
- ✅ **代码审查**：确保修复方案正确

## 常见问题 FAQ

### Q1: Logic Analyzer 与常规代码审查有什么区别？

**A**: 常规代码审查关注代码风格、命名规范、基本错误等，而 Logic Analyzer 专注于**逻辑层面的深度分析**：

| 方面 | 常规代码审查 | Logic Analyzer |
|------|-------------|----------------|
| 关注点 | 代码风格、命名、基本错误 | 逻辑正确性、状态机、并发 |
| 分析深度 | 浅层检查 | 深度分析 |
| 问题类型 | 格式、规范问题 | 逻辑错误、设计缺陷 |
| 影响范围 | 单个函数/类 | 整个系统 |

### Q2: 什么时候应该使用 Logic Analyzer？

**A**: 建议在以下场景使用：

- ✅ **关键功能开发**：涉及状态机、并发控制的代码
- ✅ **重大重构**：修改核心逻辑或架构
- ✅ **复杂变更**：多个文件、多个模块的修改
- ✅ **性能优化**：可能影响逻辑正确性的优化
- ✅ **bug修复**：修复复杂逻辑bug后的验证

### Q3: 分析报告中的问题必须全部修复吗？

**A**: 根据严重等级决定：

- 🔴 **致命问题**：必须修复，否则可能阻塞上库
- 🟠 **严重问题**：强烈建议修复，有合理理由可以放行
- 🟡 **警告问题**：建议修复，但不阻塞上库
- 🟢 **建议问题**：可选修复，作为优化方向

### Q4: 如何处理误报？

**A**: 如果发现误报（误报问题），可以：

1. **添加注释**：在代码中添加清晰的注释说明为什么这样做
2. **更新分析**：调整分析逻辑，避免类似误报
3. **记录例外**：在文档中记录已知的例外情况

### Q5: 分析需要多长时间？

**A**: 分析时间取决于变更规模：

- 小型变更（1-5个文件）：5-10分钟
- 中型变更（5-20个文件）：15-30分钟
- 大型变更（20+文件）：30-60分钟

### 作为 CI/CD 检查

```yaml
# .github/workflows/logic-analysis.yml
name: Logic Analysis

on: [pull_request]

jobs:
  analyze:
    runs-on: ubuntu-latest
```
